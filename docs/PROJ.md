# OpenClaw Embodied 产品与技术白皮书 (v2.0)

## 一、 愿景与目标

### 1. 核心命题
构建首个基于约束求解范式的消费级具身智能框架，让不具备机器人专业知识的用户，能够通过自然语言与机器人交互，并享受机器人主动服务、自主进化、技能共享的能力。
**本次调整核心**：在于防止 Agent 工作流失控（Anti-Shitstorm），通过严格区分工作流编排与动作约束，引入 **Multi-Agent 校验层**，并将视觉能力完全下沉为**文本化工具（Scene-graph）输出**。

### 2. 用户价值主张
| 当前痛点 | OpenClaw解决方案 |
| :--- | :--- |
| 机器人使用门槛高 | **一键部署**：容器化部署，无需配置底层驱动 |
| 官方APP功能固化 | **自然语言编程**：通过对话定义任务，系统自动生成可执行策略 |
| 单机能力单一 | **异构多机编排**：自动协调机械臂+底盘+轮式双臂协同完成任务 |
| 技能无法跨设备复用 | **本体无关技能共享**：技能以"约束参数"形式存在，跨品牌迁移 |
| 机器人被动响应 | **自主性与自我提升**：监听环境主动执行，空闲时自主探索避坑 |

## 二、 用户画像与场景

### 1. 目标用户分层 (Tier 体系)
| 层级 | 画像 | 设备特征 | 核心诉求 | 技术触点 |
| :--- | :--- | :--- | :--- | :--- |
| **Tier 1** | 极客/开发者 | Franka/Xlerobot/Go2 EDU | 算法验证、VLA 研究 | 全功能 SDK、ROS2、ReKep 求解器 |
| **Tier 2** | 科技爱好者 | 教育级机械臂、四足开发版 | 家务协助、编程学习 | 自然语言编程、多机编排 |
| **Tier 3** | 普通消费者 | 玩具级桌面宠物 (Nanobot) | 情感陪伴、活人感 | `SOUL.md` 个性、Heartbeat 主动唤醒 |

### 2. 核心使用场景
*   **场景A：主动服务（Proactive Service）** - 依赖 Heartbeat 机制与持久记忆，在用户未主动下发指令时，根据时间与环境触发动作（如早上9点主动泡茶）。
*   **场景B：多机协作复杂任务（Heterogeneous Task Orchestration）** - 将模糊指令分解，通过 ROS2 统一调度异构机器人的时序动作（如底盘到位后机械臂卸货）。
*   **场景C：技能自主学习与自省（Autonomous Skill Improvement）** - 深夜触发探索，验证更优的 ReKep 约束参数。若失败，记录至 `LESSONS.md` 形成避坑知识库。

## 四、 核心系统架构：基于 Markdown 的双轨多体系统

系统彻底摒弃“大模型直接控硬件”的危险黑盒，采用**“万物皆 Markdown (State-as-a-File)”**的设计理念。

### 1. “文件即状态”协议矩阵 (The Workspace API)
系统的核心是一个本地文件夹，软硬件作为独立的守护进程对文件进行读写：
*   **`EMBODIED.md` (本体声明):** 硬件层写入。定义当前连接的硬件能力（包含 SDK 到统一接口的映射）。让大脑“知道自己是谁”。
*   **`ENVIRONMENT.md` (环境感知):** 视觉 MCP 工具写入。将摄像头画面转化为纯文本的 Scene-Graph（如：*桌面[水杯, 坐标, 空] 位于 [电脑] 右侧*）。大模型不看图，只读此文件。
*   **`LESSONS.md` (经验避坑库):** 每次任务失败后的自省记录库。作为 Planner 生成动作前的强制前置上下文。
*   **`SKILL.md` (工作流记忆):** 记录经过验证的标准操作流 SOP。
*   **`ACTION.md` (软硬耦合层):** **[关键层]** 大模型写入的关键点或约束目标（ReKep格式，非代码）。底层求解器读取此文件执行。
*   **`SOUL.md` & `HEARTBEAT.md`:** 决定桌面宠物（Nanobot）的主动搭话频率与性格属性。

### 2. 软件轨 (Track A)：文本流与多体协同
不再依赖单一模型，建立标准流水线：
1.  **感知提取 MCP:** 定时拍摄画面，通过小规模 VLM 提取 Scene-Graph 覆写 `ENVIRONMENT.md`。
2.  **主控大脑 (Planner Agent):** 性能最强的基座。读取环境和指令，生成任务拆解，写出 `ACTION.md` 草稿。
3.  **动作校验官 (Critic Agent):** 独立校验模型。拦截草稿，对照 `EMBODIED.md` 物理极限进行校验，通过后才正式落盘下发。
4.  **记忆重构者 (Refactor Agent):** 夜间触发，清理 `SKILL.md` 里的冗余步骤，避免“屎山代码”。

### 3. 硬件轨 (Track B)：意图与行动对齐
1.  **ROS2 统一数据总线:** 维护传感器信息（Vision/LiDAR）、控制信息（Move/Manipulation）和交互信息的全局总线。
2.  **基于 ReKep 的约束求解引擎:** 监听 `ACTION.md` 中的几何约束指令（如“末端与杯口对齐，且Z轴向上”）。底层将这些约束转化为二次规划问题，C++ 高频实时求解出符合运动学的关节轨迹（<50ms）。
3.  **动态电子围栏系统 (Safety Geofence):**
    *   **逻辑围栏:** Critic Agent 在生成动作前，基于三维坐标预判是否越界。
    *   **物理围栏:** 结合 Franka/Elfin 的力矩传感器，在底层设置持续力（<150N）与瞬时力红线。探测到碰撞直接物理熔断，并将状态报错抛回 Track A 记录到 `LESSONS.md`。

## 五、 敏捷落地路线图 (Nanobot 先行策略)

为了在复杂硬件调试完成前快速验证整个软件协议，我们采用**从 Tier 3 到 Tier 1** 的逆向打通策略。

### Phase 1: 桌面闭环与 Markdown 协议确立 (Months 1-2)
**目标：在最简化的 Nanobot 上跑通软硬解耦与多 Agent 校验流。**
*   **Track A:** 确立 `EMBODIED.md`、`ENVIRONMENT.md` 的数据格式。完成 Planner + Critic 的双体 ReAct 循环开发。实现 Heartbeat 主动唤醒聊天。
*   **Track B:** 封装 Nanobot 的微型 SDK。开发监听 `ACTION.md` 变化并驱动电机的本地看门狗。
*   **里程碑:** 微信下发指令 -> Planner 生成动作 -> Critic 校验通过 -> Nanobot 完成点头/指向等简单交互，全程日志清晰无崩溃。

### Phase 2: 视觉解耦与工具链合并 (Months 3-4)
**目标：接管真实环境感知，沉淀避坑经验库。**
*   **MCP 与合并:** 开发真实的视觉感知 Server，将相机的多模态信息稳定降维成文本 Scene-Graph 写入 `ENVIRONMENT.md`。
*   ** Track A:** 激活 `LESSONS.md` 机制，让模型学会“吃一堑长一智”。
*   ** Track B:** 在 ROS2 环境下，跑通 Go2 EDU 四足底盘的指令下发与状态回传。

### Phase 3: 约束求解与高阶异构协同 (Months 5-6)
**目标：全面兼容工业级/科研级设备，展现 ReKep 求解与多机协同的真正威力。**
*   **Track B (核心发力):** 基于 Franka 和 Xlerobot，完成 C++ 版本的高性能 ReKep 约束求解器并接入 ROSClaw Bridge。
*   **Track A:** 升级调度逻辑，在 `ACTION.md` 中实现多设备并发指令的时间锁/空间锁。
*   **里程碑:** 实现从“桌面 Nanobot 情感交互”到“一车一臂（Xlerobot+底盘）协同整理客厅”的跨越，并正式上线基于 `SKILL.md` 的社区生态市场。
